## CI Failure Diagnosis

PR #18 initial CI failed on `Tests (ubuntu-latest, Python 3.11)`:

### Failing test
`tests/e2e/test_standard_workflow.py::test_full_standard_workflow_survives_restart_and_reproves_after_edit`

### Error
```
hardproof.errors.TransitionError: 1 fresh passing verification check required; found 0
```

### Root cause

`normalize_timestamp` in `hardproof/domain/models.py` used second-precision (`timespec="seconds"`). The e2e test performs two rapid verification calls:

1. First `evidence.verify(run_id)` — creates PASSED evidence (workspace state A)
2. Modify file — makes first evidence STALE
3. Second `evidence.verify(run_id)` — creates PASSED evidence (workspace state B)
4. `passing_check_names()` sorts by `(completed_at, id)` to find the latest evidence per check name

When both verifications complete within the same second (common on fast CI runners), both records have the same `completed_at`. The tie-breaking falls to `id` (a random UUID). If by chance the older (STALE) record's UUID sorts after the newer (PASSED) record's UUID, `passing_check_names` sees the STALE record as "latest" and returns zero fresh evidence passes.

### Fix

Changed `normalize_timestamp` from `timespec="seconds"` to `timespec="milliseconds"`. This ensures rapid verify calls produce distinct timestamps, so `passing_check_names` always orders by actual wall-clock time rather than random UUID.

### Regression test

Added `test_latest_pass_overrides_stale_for_same_check_name` in `tests/unit/test_evidence_service.py`.

### Cross-platform analysis

- **Ubuntu Python 3.11**: FAILED (fast runner, same-second timing)
- **Ubuntu Python 3.12**: PASSED (different timing characteristics)
- **macOS Python 3.11**: PASSED (slower subprocess overhead)
- **Windows Python 3.11**: PASSED (slowest subprocess overhead)

The failure was platform/timing-dependent, not Python-version-dependent. The millisecond-precision fix resolves it on all platforms.
