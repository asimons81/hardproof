## Stage-graph evidence

- Configuration compiles into a finite single-successor path over the fixed Hardproof vocabulary.
- Explicit limits are 12 vocabulary nodes and 24 configured edges; unknown stages, duplicates,
  self-loops, backward edges, ambiguous successors, unreachable requirements, and paths without
  VERIFY/DELIVER/COMPLETE fail closed.
- Only DISCOVERY and LEARN may be skipped; Critical overlays cannot skip either. Profile overlays
  validate independently and are selected from the run profile at evaluation time.
- `config explain` reports the source, profile, graph, required/skipped stages, active packs,
  immutable constraints, and stable configuration fingerprint without configuration values.

## Policy-pack evidence

- Versioned data-only Python, Node, Rust, and Go packs provide deterministic command categories,
  local repository indicators, and suggested checks without subprocess, network, or code evaluation.
- Pack matches use stable `pack.<language>.<category>` rule IDs and report pack version `1.0`.
- Tests cover test/lint/build/generate/publish commands, Windows executable suffixes, detection order,
  and elevated publication risk. Existing wrapper and chain normalization runs before pack matching.

## Migration 002 evidence

- Adds append-ordered policy decision traces, waiver state and lifecycle events, and risk suggestions without altering v1 tables.
- Fresh databases apply migrations 001 and 002; real schema-v1 run and evidence records survive upgrade unchanged.
- Reopen/idempotence, incomplete/failed migration rollback, newer-schema refusal, and package-data inclusion pass.
- Model validation and SQLite constraints independently reject immutable-rule waivers.
- YAML config schema remains v1 until its separately tested Task 3 migration; database and config version constants are no longer conflated.

## Terminal normalization evidence

- Pure bounded tokenizer splits POSIX/PowerShell chains and unwraps `sudo`, `env`, `command`, `cmd`, PowerShell, `pwsh`, `bash`, and `sh` launchers without execution or expansion.
- Immutable force push, destructive Git/filesystem/permission actions, deployment/publication, database migration, credential access, tests, builds, and inspection receive deterministic categories and stable keys.
- Highest-risk chained segment governs policy; safe prefixes cannot hide an immutable or destructive suffix.
- Empty, malformed, overlong, over-segmented, and over-tokenized input is safely classified as ambiguous without echoing command content.
- Windows path separators remain intact, and property-generated bounded Unicode input always returns a structurally valid result.

## Configuration v2 evidence

- Missing configs use schema v2 defaults; explicit schema-v1 files migrate only in memory and retain their reported source version.
- Equivalent v1 and v2 effective configurations produce the same deterministic SHA-256 fingerprint.
- Named `project.*` allow, deny, and approval rules validate tool selectors, bounded conservative regexes, project-relative globs, profile/stage scopes, and mandatory rationale.
- Unknown keys/effects/profiles/stages/packs, duplicate rule keys/packs, traversal, nested-quantifier/lookaround/backreference regexes, and open failure mode with a Critical default fail closed with field locations.
- Only `artifact_directory` expands non-secret environment variables; policy regexes and globs remain literal data.
- Config initialization emits schema v2, while no existing user config is rewritten automatically.

## Traceable policy engine evidence

- One evaluator enforces immutable, project deny, project approval, stage/core, project allow, and default behavior with stable final keys.
- Immutable rules short-circuit project configuration; deny wins over approval regardless of YAML order; project allow cannot bypass a stage source-mutation block.
- Every evaluator result ends with a trace entry matching its final key and round-trips through the public JSON-friendly domain contract.
- Hooks persist allow, block, and approval decisions with ordered traces, configuration fingerprints, argument digests, and no raw arguments or secret-bearing command text.
- Standard/Critical mutations fail closed if required policy-ledger persistence is unavailable; existing public Hermes approval directives remain compatible.
- Failure behavior supports `profile`, `open`, and `closed`, while Critical mutation never fails open.

## Human waiver evidence

- Only attributable CLI, slash, gateway, desktop, or supported chat actors can create or revoke policy waivers; model/tool and known nonhuman actors are refused before storage.
- Each waiver requires a unique name, exact rule key, rationale, creation/expiry times, and optional run/tool/command-digest/path/profile/stage scopes that match by intersection.
- `terminal.immutable.*`, state, evidence, verification, migration, and approval-authenticity namespaces are rejected at creation and again by the pure matcher.
- Active exact scopes convert only their matching block/approval to an `allow` decision whose final trace outcome is `waived` and whose durable record contains the waiver ID.
- Revocation and automatic expiry append one lifecycle event, survive restart, and make matching impossible at the supplied time boundary.
- Creation/revocation rationale is redacted before durable storage; list output omits rationale and raw command content.
- Human CLI and slash adapters share `policy waivers create|list|revoke`; model-callable command contexts can list but cannot mutate.

## Policy explanation evidence

- Durable policy decisions expose append sequence numbers and can be fetched only within the active run.
- `policy explain --event <sequence>` reproduces stored action, final key, safe reason, ordered trace, argument/config hashes, waiver identity, and creation metadata.
- `policy explain --tool <name> --args-json <object>` evaluates current validated config/run/waiver state without executing a command or appending policy/lifecycle records.
- Human output is compact and ordered; `--format json` is canonical, sorted, and reproducible.
- Hypothetical JSON is object-only and size-bounded before parsing; output contains digests rather than raw arguments.
- The existing sixth `hardproof_report` tool adds read-only `policy_explain` access, preserving the exact six-tool public surface and exposing no waiver mutation action.
- Registered-plugin integration proves secret-bearing hypothetical arguments do not appear in tool responses.

## Risk suggestion evidence

- A deterministic local classifier evaluates bounded task text, normalized file paths, and the terminal classifier; it performs no model, subprocess, network, or global-state access.
- Authentication, secrets, billing, concurrency, migration/infrastructure, deployment/destructive/database/credential commands, public APIs, configuration, multi-file scope, and documentation-only work emit categorical levels with stable reason codes.
- Suggestions persist separately from task-selected risk and never change a run profile; task creation returns both values and marks the human decision pending.
- Status reports pending suggestion count. Human `policy risk decide` records acceptance or override with actor, source, time, and mandatory override rationale.
- Nonhuman decision attempts and repeated decisions are refused; stored rationale is redacted.
- `policy suggest-risk` persists a human workflow record, while read-only `hardproof_report(action=risk_suggest)` is pure and leaves both ledgers and profile unchanged.
