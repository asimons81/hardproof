# Hardproof v0.1.1 Release Recovery Plan

**Created**: 2026-07-11
**Reason**: v0.1.0 tag is unsigned. Release workflow requires `git tag -v` to pass.

## Why v0.1.0 will not be retagged

The `v0.1.0` tag (`b419a40d`) is an annotated tag pointing at the audited release commit `dd47edb`. It was published to GitHub and a GitHub release was created with attached artifacts. Moving or replacing it would:

- Break downstream references to the immutable tag
- Invalidate the existing GitHub release
- Violate Hardproof's commitment to tag immutability
- Set a bad precedent for future releases

Instead, v0.1.1 is cut as a release-infrastructure patch with identical application behavior.

## Signing backend: SSH (Ed25519)

- SSH key `id_ed25519` already exists on Tony's machine
- No GPG key infrastructure exists
- Git 2.34+ natively supports SSH signing
- CI verification requires only the public key (no secret material in CI)
- GitHub supports SSH signing keys

### CI verification design

Before `git tag -v`, the workflow configures:

```bash
git config gpg.format ssh
git config gpg.ssh.allowedSignersFile "$GITHUB_WORKSPACE/.github/release-signers"
```

`git tag -v` then verifies the tag's SSH signature against the allowed signer.

### Local signing configuration

```bash
git config gpg.format ssh
git config user.signingkey "/c/Users/asimo/.ssh/id_ed25519"
git config user.email "asimons81@gmail.com"
```

Tag creation:

```bash
git tag -s v0.1.1 <commit> -m "Hardproof v0.1.1 Public Alpha"
```

`git tag -s` with `gpg.format=ssh` creates an SSH-signed tag.

## Expected v0.1.1 release process

1. Create `fix/v0.1.1-release-signing` branch from `main` (`0a0db26`)
2. Add SSH signing infrastructure and workflow repair
3. Bump version to 0.1.1
4. Run full verification suite (217+ tests, lint, mypy, build, smoke installs)
5. Open PR into `main`
6. Merge after CI passes
7. Create signed tag `v0.1.1` with SSH signature
8. Push tag → triggers release workflow
9. Workflow verifies signature, runs gates, publishes to PyPI via Trusted Publishing
10. Verify PyPI, clean-install smoke test
11. Open documentation PR

## Rollback and failure behavior

- If signing fails locally: diagnose key availability, retry with SSH agent if needed
- If `git tag -v` fails in CI: allowed-signers file or key mismatch; fix and push new tag
- If PyPI publication fails after GitHub release: artifacts remain on GitHub; diagnose and re-push tag
- v0.1.0 is never touched: it's the fallback GitHub release
- No application behavior changes: rollback is just using the v0.1.0 release

## Non-negotiable constraints

- v0.1.0 tag: immutable, untouched
- Release commit dd47edb: immutable
- Signed-tag verification: preserved and repaired, not weakened
- PyPI: Trusted Publishing only, no static tokens
- No v0.2.0 or Gatehouse code: v0.1.1 is a pure infrastructure patch
