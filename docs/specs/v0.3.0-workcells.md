# v0.3.0 Workcells Specification

Status: Released (v0.3.0). Hardened by v0.3.1 Workcells Hardening.

## Outcome

Workcells executes an approved implementation plan through fresh Hermes child
sessions using a durable task graph, deterministic waves, bounded context,
validated child results, bounded retries, and resumable orchestration. It
preserves human authority, existing profiles/stages, fresh run-level
verification, the six-tool surface, and forward-only state migration.

## Graphs, revisions, and waves

A graph belongs to one run and records an approved-plan artifact hash for
Standard and Critical profiles. A task has a stable filename-safe key,
objective, acceptance criteria, required flag, dependencies, read/write
scopes, priority, model tier, bounds, artifact paths, graph revision, and
status. Graph validation rejects cycles, self or duplicate edges, unknown
dependencies, active/completed-task rewrites, and ambiguous required paths.
Material changes create a new revision and require renewed plan approval where
the profile requires it.

Ready means every required dependency succeeded. Deterministic waves sort ready
tasks by priority then key. Failed, blocked, cancelled, interrupted, or
unresolved dependencies block their dependents. Quick permits one compact task;
Standard and Critical require plan approval before launch.

## Attempts, claims, and child execution

Each task attempt has a unique ID, immutable terminal result, selected tier,
context hash, launch token, lifecycle events, and optional Hermes child handle.
An atomic SQLite claim transitions only a ready task to starting. It prevents
duplicate launch, permits one active attempt, and never silently steals an
abandoned claim.

The internal adapter calls public `PluginContext.dispatch_tool` for
`delegate_task`, with an explicit brief and bounded context. It records launch
output and child identity where returned. Public Hermes does not supply durable
child status, wait, cancellation, or reconnect; reconciliation therefore marks
unobservable attempts interrupted/recovery-required, blocks dependent work, and
requires explicit retry or escalation. No private Hermes state is used.

## Artifacts and result contract

Artifacts are inside `.hardproof/runs/<run>/tasks/<task>/attempts/<n>/` and are
atomically written after path, symlink/junction, size, and redaction checks.
`brief.md` and deterministic `context.json` contain only explicit bounded
inputs—never a parent transcript or hidden reasoning. A versioned `result.json`
contains the run/task/attempt/child identities, reported outcome, bounded
summary, changed paths, commands/tests/outcomes, artifacts, blockers, evidence
references, and recommended action.

The parent validates identity, version, size, paths, redaction, artifact
existence, active-attempt ownership, and task acceptance before making the
authoritative transition. Results are untrusted and are not fresh verification
evidence.

## Retries, tiers, recovery, and workspace safety

Maximum attempts are bounded by validated configuration. A retry requires an
attributed material change to the brief, scope, dependency context, human
guidance, tier, acceptance criteria, or failure remediation; identical retries
are refused. Exhaustion blocks or escalates the task.

Tiers are provider-neutral `economy`, `standard`, and `strong` configuration
keys. Profile minima and deterministic retry escalation govern selection.
Default shared-workspace scheduling allows one mutating child. Optional
parallelism requires explicit configuration, clean expected state, known
non-overlapping write scopes, and a small bound. Workcells does not claim
filesystem isolation.

## Boundaries

No seventh tool, independent reviewer system, worktree isolation, execution
backend, dashboard, learning extraction, telemetry, account, or hidden network
behavior is part of v0.3.0. COMPLETE still requires fresh successful evidence;
Critical completion still requires human approval.
