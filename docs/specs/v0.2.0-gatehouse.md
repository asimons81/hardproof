# v0.2.0 Gatehouse Specification

Status: Implemented release-candidate specification; independent audit pending

## Outcome

Gatehouse makes every policy result explainable and safely configurable without editing Python. A user can reconstruct why an action was allowed, blocked, approval-gated, or waived from durable records, while immutable safety rules remain impossible to waive or override.

## Compatibility and boundaries

- Preserve the v0.1.1 public Hermes plugin key, entry point, six tool names, slash command, CLI command, hooks, and nine namespaced skills.
- Preserve project-local operation, no telemetry, no routine network traffic, public Hermes APIs only, and minimum Python 3.11.
- Migrate v0.1.0 databases forward transactionally. Never downgrade, rewrite, or delete v0.1.0 records.
- Configuration is strict YAML with explicit schema versions. Unknown keys and unsafe combinations fail with path-aware diagnostics.
- Policy evaluation is pure and deterministic for a normalized input, configuration, run state, and clock value supplied by the caller.

## Policy decision and trace

Every evaluation returns:

- final action: `allow`, `block`, or `approval`;
- stable final rule key and human-readable reason;
- ordered trace entries containing rule key, outcome (`matched`, `not_matched`, `superseded`, `waived`), and redacted explanation;
- optional named waiver identifier;
- normalized risk suggestion and reasons;
- a deterministic policy/config fingerprint.

Trace ordering is fixed: state availability, immutable terminal safety, project deny, project approval, stage mutation, project allow, policy pack, default. Raw secrets, full command content, and unbounded arguments are not persisted.

## Terminal classification

Tokenize terminal input without executing or shell-expanding it. Recognize command chains and common wrappers across POSIX and PowerShell syntax. Classify at least:

- immutable force push;
- destructive Git reset/clean/branch deletion;
- recursive filesystem deletion;
- permission/ownership mutation;
- package publication and deployment;
- database migration/destruction;
- secret-oriented and credential commands;
- read-only inspection, test, lint, build, and language-tool commands.

Malformed or ambiguous input receives a conservative traceable classification. Parsing is a policy aid, not a security sandbox or a complete shell implementation.

## Project rules

Configuration supports ordered named terminal/tool rules with:

- stable project-local key;
- `allow`, `deny`, or `approval` effect;
- tool selector;
- optional command regex and path glob;
- optional profiles and stages;
- non-empty rationale.

Regexes are size-bounded and compiled during config validation. Project rules cannot replace `terminal.immutable.*` rules. Deny wins over approval; approval wins over allow. An allow rule cannot bypass stage mutation policy unless that rule explicitly targets an artifact path already allowed by the core policy.

## Waivers

Waivers are durable records created only through human CLI, slash, gateway, desktop, or supported chat surfaces. Each waiver has:

- unique name and identifier;
- exact stable rule key;
- optional tool, command-digest, path, profile, stage, and run scope;
- mandatory rationale and creator/source;
- creation and expiry timestamps;
- active/revoked state and revocation audit data.

Waivers require an expiry, match by intersection of every supplied scope, and never match `terminal.immutable.*`, state corruption/unavailability, evidence freshness, human approval authenticity, or migration safety rules. Model-callable tools may list and explain waivers but may not create, extend, or revoke them.

## Failure mode

`policy.state_failure_mode` is `profile` (default), `open`, or `closed`. `profile` preserves v0.1.0 behavior. `open` is rejected for Critical runs and never applies to immutable rules. Every state failure records the selected behavior in its trace.

## Risk suggestions

Risk suggestions are deterministic advisory values (`low`, `medium`, `high`, `critical`) derived from file paths, terminal categories, task text, repository policy packs, and configured signals. Suggestions never silently change a run profile or create an approval. The user can accept or override a suggestion with a recorded rationale.

## Stage graph

Projects may:

- disable only optional discovery or learning stages for Quick/Standard profiles with a recorded skip policy;
- add named required artifacts or human approval gates to supported forward edges;
- prohibit selected forward edges for selected profiles.

Projects may not reorder stages, create cycles, bypass VERIFY or DELIVER, remove Standard design/plan approval, weaken Critical requirements, or make terminal states mutable. Validation returns the exact unsafe edge and reason.

## Policy packs

Ship original built-in packs for Python, Node, Rust, and Go. Packs provide repository detection, read-only/build/test command categories, risk path signals, and suggested verification checks. Explicit configuration wins only where doing so does not weaken immutable policy. Detection reads local files only and records which indicators matched.

## Commands and diagnostics

Add:

- `hardproof policy explain [--event SEQUENCE | --tool NAME --args-json JSON]`;
- `hardproof policy waivers list`;
- human-only `hardproof policy waivers create` and `revoke`;
- `hardproof policy suggest-risk`;
- richer `hardproof config validate`, `hardproof doctor`, and `hardproof db migrate --dry-run` output.

Slash and Hermes CLI surfaces must expose equivalent safe operations. Explain output is deterministic JSON plus a compact human form.

## Storage migration v2

Migration 002 adds append-only policy decision/trace storage, waiver lifecycle records, config fingerprints, and risk suggestion/override records. Foreign keys, timestamp formats, uniqueness, expiry, and immutable-rule constraints are enforced in both service validation and database checks where SQLite can express them.

Migration evidence must cover fresh v2 creation, v1-to-v2 upgrade, interrupted rollback, reopen/idempotence, newer-schema refusal, and preservation of v1 run/evidence/report data.

## Security and privacy gates

- Immutable rules are exhaustively proven unwaivable under generated scopes.
- Only authenticated human surfaces mutate waiver state.
- Regex, glob, trace, and persisted previews are bounded.
- Policy logs contain hashes and redacted summaries, never raw secret-bearing arguments.
- Configuration cannot introduce executable Python, remote includes, environment-secret interpolation, or network fetches.
- Policy packs contain data and deterministic Python logic only; no subprocess or network execution during detection.

## Acceptance criteria

- Every allowed, blocked, approval-gated, and waived action returns and persists an ordered explainable trace.
- Stable rule keys remain compatible with v0.1.0 and are documented.
- Users can define validated allow, deny, and approval rules in YAML.
- Human users can create, inspect, expire, and revoke precisely scoped waivers; immutable rules remain unwaivable.
- Failure behavior is configurable within the Critical/immutable constraints.
- Risk suggestions are explainable and never silently authoritative.
- Supported stage-graph changes validate; every weakening or cyclic change fails closed.
- Python, Node, Rust, and Go packs detect representative repositories and suggest correct checks.
- v1 databases upgrade to v2 without losing any record; failed migration rolls back.
- CLI, slash, gateway registration, restart, local-only, redaction, package, and cross-platform CI contracts pass.
- Documentation includes configuration reference, rule-key catalog, waiver threat model, migration guide, and worked explanation examples.
- A release report maps every criterion to current test, migration, security, compatibility, documentation, and package evidence.
